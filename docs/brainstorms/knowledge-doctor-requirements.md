# ps:knowledge-doctor 需求文档

**日期**：2026-04-16
**版本**：v0.2
**状态**：设计已拍板，待实现

## 1. 目标

让用户随时知道知识库**缺什么、够不够用、去哪找**。从被动等用户喂数据，变成主动告诉用户"你的知识库差什么，找谁要"。

## 2. 问题与动机

当前知识库建设完全靠用户主动：用户不知道完整知识库长什么样、不知道手边的材料该怎么喂、不知道喂完了还差多少。结果是知识库要么空着，要么喂了一半不知道缺什么。`rfp-analyze` 的 Go/No-Go 判断因此基于不完整数据，用户对结果也没信心。

## 3. 核心设计

### 3.1 完整度模型

对照知识库固定目录结构，定义"最低可用"基线：

| 维度 | 最低基线 | 充足标准 | 数据位置 |
|------|---------|---------|---------|
| 公司资质 | ≥5 条有效（未过期）资质 | ≥15 条 | `company-profile.yaml.qualifications[]` |
| 团队汇总 | roster.yaml 存在 + cert_summary 非空 | ≥100 条有效证书 | `团队/roster.yaml` |
| 团队明细 | ≥1 个 cert-registry 分片 | ≥3 个分片 | `团队/cert-registry-*.yaml` |
| 产品档案 | ≥1 个产品 YAML | ≥3 个 | `产品档案/*.yaml`（排除 example） |
| 竞品对比 | ≥3 家竞品 YAML | ≥10 家 | `竞品/*.yaml`（排除 README） |
| 公司介绍 | ≥1 个文件（营业执照 or 简介） | ≥3 个 | `公司介绍/*`（排除 README） |
| 客户案例 | ≥3 个文件 | ≥10 个 | `客户案例/*`（排除 README） |
| 案例引用 | ≥1 条 case_references | ≥5 条 | `company-profile.yaml.case_references[]` |
| 差异化亮点 | ≥1 条 highlights | ≥3 条 | `company-profile.yaml.highlights[]` |

### 3.2 诊断报告格式

```
📊 知识库健康度报告
━━━━━━━━━━━━━━━━━━
整体完整度: 65% (6/9 维度达标)

✅ 充足 (3)
  公司资质    29 条（10 有效 / 19 过期）
  团队汇总    1085 条有效证书，764 人
  竞品对比    29 家

⚠️ 不足 (3)
  公司资质有效率  仅 34%（19 条已过期）
    → 找法务部/行政部要最新证书扫描件，优先续期 ISO27001 和 CCRC 系列
  客户案例       19 个图片素材，0 条结构化案例
    → 找售前团队/项目经理整理最近 3-5 个成功案例，需要：客户名/行业/金额/摘要
  差异化亮点     0 条
    → 从护网战绩和资质优势中提炼 3-5 条可量化亮点

❌ 空缺 (3)
  产品档案      0 个产品有结构化档案
    → 找产品线负责人要 datasheet / 产品手册，或从官网下载
    → 每个产品建一个 YAML：知识库/产品档案/{product-slug}.yaml
  案例引用      company-profile.yaml 中 case_references 为空
    → 客户案例入库后，在 company-profile.yaml 加引用条目
  团队花名册    有杭州本地团队，无全公司核心人员
    → 找 HR 要核心项目人员简历（项目经理/架构师/安全专家）

下次入库建议:
  优先级 1: 产品档案（bid-draft 技术方案章节的必要输入）
  优先级 2: 结构化客户案例（rfp-analyze 案例匹配的必要输入）
  优先级 3: 续期过期证书（当前 19 条过期影响 Go/No-Go 判断）
```

### 3.3 指导粒度

每个缺口项提供：
- **现状**：有什么、有多少
- **差距**：距离基线差多少
- **去哪找**：建议找谁（角色/部门）、从哪个系统获取
- **怎么喂**：入库命令或操作步骤（如 "把 PDF 扔进 知识库/资质证书/ 后跑 /ps:knowledge-ingest certs"）

## 4. 使用场景

### 4.1 独立诊断
```
用户: /ps:knowledge-doctor
→ 输出完整健康度报告
```

### 4.2 入库后自动诊断
```
用户: /ps:knowledge-ingest <path>
→ 入库完成
→ 自动追加简版健康度摘要（只显示变化项 + 仍缺项）
```

### 4.3 投标前快速检查
```
用户: /ps:rfp-analyze <slug>
→ rfp-analyze 开始前自动检查知识库是否足够支撑本次分析
→ 若关键维度空缺，先警告再继续
```

## 5. 范围

### 5.1 纳入
- `ps:knowledge-doctor` 独立 skill（完整报告）
- 入库后 mini 报告（集成到 `ps:knowledge-ingest` Phase 4 结果汇报后）
- 完整度模型定义（基线 + 充足阈值）
- `scripts/ps_knowledge_doctor.py` — 扫描知识库目录 + 读取 YAML + 输出 JSON 诊断结果
- 缺口建议模板（每个维度一条"去哪找"建议）

### 5.2 非纳入
- 自动从外部系统抓取数据（官网/OA/CRM）
- 证书到期提醒推送（可叠加但不包含在本期）
- 知识库内容质量评估（如"这个案例描述质量低"）— 只做数量+结构检查
- rfp-analyze 的集成（场景 4.3 放 v0.3）

## 6. 验收标准

1. 对当前已有数据的知识库跑 doctor → 报告正确识别 3 个充足 + 3 个不足 + 3 个空缺
2. 完全空的知识库（刚 setup 完）跑 doctor → 全部标为空缺，每项都有"去哪找"建议
3. 入库后 mini 报告只显示本次变化 + 剩余缺口，不重复完整报告
4. 报告中的"怎么喂"指令可直接复制执行

## 7. 与路线图的关系

- **替代 skill-catalog 中的 `ps:status`**：doctor 比 status 更聚焦（只管知识库健康度，不管商机状态）
- **为统一入库铺路**：doctor 的缺口列表就是入库优先级指南，用户按建议补材料 → 再跑 ingest → 再跑 doctor = 闭环
- **为 rfp-analyze 增信**：未来 rfp-analyze 开头先跑 doctor，用户看到"知识库 85% 完整度"才会信任分析结果
